* §0 motivation

** 0.1 既 SHIPPED 部分

Phase 2.A "native menu bar (GMenuModel)" は CLAUDE.md の phase plan で
*未* 扱いだったが、実は infrastructure 全体が既 SHIPPED 済:

- *Rust 側* (`src/gtk_backend.rs`):
  - `nelisp-gtk-set-menu-bar SPEC` extern (= GMenuModel build + GtkApplication 結線)
  - `nelisp-gtk-poll-menu-event` extern (= action string queue で elisp に dispatch)
  - `nelisp-gtk-show-context-menu` extern (= 右クリック context menu、Phase 2.S 済)
  - `menu_event_queue: VecDeque<String>` (= 内部 dispatch 機構)

- *Elisp 側* (`nemacs-gtk-frontend.el`):
  - `nemacs-gtk--menu-spec` defconst (= File / Edit / Help 3 menu)
  - boot 時 `(nelisp-gtk-set-menu-bar nemacs-gtk--menu-spec)` 呼出
  - `nemacs-gtk--handle-menu-action` dispatcher (= "save" / "quit" / "cut" 等の string action を command に変換)

つまり menu bar **可視 + click 可動作 はすでに使える**。CLAUDE.md phase plan
での "Phase 2.A 完了" gate (= File / Edit menus visible) は満たしている。

** 0.2 v0.1 残課題 (= 本 doc の scope)

ただし以下が user 体感で未完:

1. *M1 commands との wiring 不足* — File menu の "Open" / "Save" 等が
   M1 の =find-file= / =save-buffer= / =dired= を直接呼ばない (= 旧
   `nemacs-gtk-keyboard-find-file` legacy handler 経由)
2. *M2 commands の menu 露出なし* — describe-function / undo / isearch 等
   M2 で ship したものが menu に出ていない
3. *accelerator hint 未表示* — menu 項目の右に "Ctrl+O" 等の表示なし
   (GMenuModel の accel 結線必要)
4. *buffer-list submenu の dynamic 化* — 既 `nemacs-gtk--buffer-menu-spec`
   関数あるが menu bar に組み込まれてない (= context menu のみ)

- M2 dispatch: Edit menu → =isearch-forward= / =undo= / =undo-redo= / =eval-defun= に直結、Help menu → =describe-function= / =describe-variable= / =describe-key= / =M-x ielm=

* §2 architecture

** 2.1 既 menu_event_queue 機構を維持

dispatch flow (= 変更なし):

```
User clicks "File" → "Open"
  → GtkApplication action triggered
  → Rust pushes "open-file" string into menu_event_queue
  → elisp main loop polls (nelisp-gtk-poll-menu-event) → "open-file"
  → nemacs-gtk--handle-menu-action "open-file" dispatched
  → (call-interactively #'find-file)
```

本 phase は (a) menu spec の枝を増やす、(b) action string → command の
mapping を増やす、の 2 点のみ。Rust 側変更は accelerator 表示分のみ。

** 2.3 separator + accelerator extension

separator の表現は現状 `("---" . nil)` で codex に解釈してもらう (= Rust
側が `gio::Menu::insert_section` で扱う)。

accelerator 表示は GMenu の `accel` attribute で結線。Rust 側 patch:

```rust
// src/gtk_backend.rs install_menu_bar / build_submenu
//   - For each leaf, if action string has a known accel mapping,
//     call menu.set_attribute_value("accel", ...).
```

accel mapping は elisp 側 defvar から渡す:

```elisp
(defconst nemacs-gtk--menu-accels
  '(("find-file"            . "<Ctrl>X <Ctrl>F")
    ("save-buffer"          . "<Ctrl>X <Ctrl>S")
    ("dired"                . "<Ctrl>X D")
    ("switch-to-buffer"     . "<Ctrl>X B")
    ("kill-buffer"          . "<Ctrl>X K")
    ("undo"                 . "<Ctrl>/")
    ("undo-redo"            . "<Ctrl>X U")
    ("isearch-forward"      . "<Ctrl>S")
    ("isearch-backward"     . "<Ctrl>R")
    ("eval-last-sexp"       . "<Ctrl>X <Ctrl>E")
    ("eval-region"          . nil) ; not bound
    ("describe-function"    . "<Ctrl>H F")
    ("describe-variable"    . "<Ctrl>H V")
    ("describe-key"         . "<Ctrl>H K")))
```

new extern: `(nelisp-gtk-set-menu-accels ALIST)` で Rust にこの map を
事前共有、`nelisp-gtk-set-menu-bar` 後の build phase で参照。

* §6 open question

** 6.1 既存 nemacs-gtk-keyboard-find-file との重複

`nemacs-gtk-frontend.el` line 440 に既存の =nemacs-gtk-keyboard-find-file= 
function あり。M1 の =find-file= (=emacs-fileio.el=) と命名が違うので
直接の collision はない。但し:

- 旧: =nemacs-gtk-keyboard-find-file= = GTK file dialog (= GtkFileChooser) を pop
- 新: =find-file= (M1) = minibuffer prompt + read-file-name

UX として menu の "Open File..." はどちらを呼ぶべきか?
- 案 X: GTK file dialog (= UI 統一)
- 案 Y: minibuffer (= Emacs 互換)

決定 = case 案 X (GTK 標準体験) を default、=C-x C-f= keybinding は minibuffer
(= M1 互換) で残す。両者共存。

** 6.2 menu_event_queue の throughput

現在の `(nelisp-gtk-poll-menu-event)` は main loop 1 tick/event。複数 menu
click 連打で queue 詰まりは想定されないが、Phase 2.E (multi-frame) 後に
window 単位 queue が必要になる可能性。本 phase では single queue 維持。

** 6.3 accelerator string の GTK 形式

GTK の accel string は `<Ctrl>X <Ctrl>F` 形式 (= GtkApplication.set_accels_for_action)。
elisp 側 keybind と整合した string 生成 helper が必要。
=nemacs-gtk--keybind-to-gtk-accel= を Task B で実装。

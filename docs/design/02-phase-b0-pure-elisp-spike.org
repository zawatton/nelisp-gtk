* §0 motivation

The =nelisp-gtk= README "Long-term goal — pure-elisp endgame" section
sets B0 / B / C as the path from the current Rust-binary + elisp-policy
split to a pure-elisp GTK4 frontend running on a NeLisp runtime.  Doc 02
locks the scope of *Phase B0* — the first spike that proves elisp can
drive GTK4 end-to-end without any Rust application logic.

The point of a spike (not a feature) is to *de-risk* the unknowns:

- Can elisp register a callback that GTK invokes from inside its main
  loop?
- Can the GLib main loop be advanced from elisp without blocking
  NeLisp's other I/O?
- How does GObject ref counting interact with NeLisp GC?

Once those answers exist, Phase B (= replace gtk4-rs entirely) becomes
an incremental porting effort instead of a research project.

* §1 success criteria

The spike is *complete* when:

1. =cargo build= produces nothing — the spike entry point is a pure
   =nelisp= invocation of an elisp file, e.g.
   #+begin_src sh
   nelisp -l src/nelisp-gtk-ffi.el -l examples/hello-window.el
   #+end_src
2. Running the above opens a GTK4 OS window titled =nelisp-gtk B0 spike=.
3. The window contains a =GtkDrawingArea= that fires a =draw= signal
   when exposed.
4. The =draw= signal callback is an elisp function.  It calls Cairo
   primitives (= =cairo_set_source_rgb=, =cairo_rectangle=,
   =cairo_fill=) via FFI to paint a coloured rectangle.
5. Closing the window via the WM close button exits the elisp process
   cleanly (= no segfault, no leaked file descriptors).
6. =examples/hello-window.el= is *under 200 elisp LOC*.  If the
   reference impl needs more than that, the FFI surface in
   =src/nelisp-gtk-ffi.el= is doing the wrong job.

Non-goals for this spike:

- Multi-window support
- Keyboard / mouse input handling (= deferred to Phase B0.c if ever)
- Pango text rendering (= deferred to Phase B0.d)
- Replacement of any current Rust code in =src/= (= that is Phase B)
- Performance benchmarks (= advisory only at the end)

* §3 prerequisite — `nl-ffi-closure` (Phase B0.a)

The current =nl-ffi-call= in =nelisp/build-tool/src/eval/ffi.rs= (820
LOC) is forward-only.  It uses the =libffi= crate's =Cif= but never
invokes =Closure=.  We need two new primitives:

** 3.1 `nl-ffi-closure-new`

#+begin_src elisp
(nl-ffi-closure-new SIG LAMBDA)
  SIG    = vector of type keywords (same shape as `nl-ffi-call').
  LAMBDA = elisp callable; receives ARGS as a list, must return a
           value of the C return type indicated by SIG[0].

  Returns: opaque integer (= C function pointer cast to uintptr_t)
  that can be passed to nl-ffi-call as a `:pointer' argument.
#+end_src

Rust implementation outline:

#+begin_src rust
// One closure is heap-allocated and leaked into a registry so the
// trampoline always finds it.  Dropping is via nl-ffi-closure-free.
struct ClosureEntry {
    cif: Cif,
    closure: Closure<'static>,
    callback: Box<dyn Fn(Vec<Sexp>) -> Sexp + Send>,
}
// The libffi trampoline (= extern "C" fn(...)) takes the raw arg
// pointers, marshals them via the stored Cif to Sexp, calls the
// elisp lambda through the captured Env handle, marshals the
// returned Sexp back to the C return slot.
#+end_src

** 3.2 `nl-ffi-closure-free`

Releases the closure (= drops the registry entry).  Optional for the
PoC — the spike can leak the single =draw= callback for its lifetime
without functional impact, but the API should exist for hygiene.

** 3.3 risks

- *Thread safety*: GTK callbacks run on the main thread.  Our elisp
  runs on the main thread.  No threading.  But this MUST be enforced
  in the closure trampoline — if any GTK API ever calls back from a
  non-main thread (= some GIO async patterns do), the trampoline must
  reject or queue, not invoke elisp re-entrantly.
- *Re-entrancy*: GTK can call our callback while our elisp is inside
  another FFI call.  NeLisp's eval stack must tolerate nested elisp
  invocations.  This is normal elisp behaviour (= advice / hooks) so
  expected to work, but document it.
- *Error propagation*: If the elisp callback throws, the trampoline
  must catch and return a default C value (= 0 / NULL / false) plus
  log the error.  Letting an elisp non-local exit unwind through
  libffi's frame is undefined behaviour.

* §6 exit gate

Phase B0 is *MERGED-OK* when:

1. `nelisp` main has `nl-ffi-closure-{new,free}` shipped + ERT tests
   covering "elisp callback runs and returns correct C type".
2. `nelisp-gtk` `feat/phase-b0-pure-elisp-spike` branch contains:
   - `src/nelisp-gtk-ffi.el`
   - `examples/hello-window.el`
   - a brief findings note in this doc (= §7 to be filled in at exit)
3. A screen recording (= attached to the feat branch merge PR) shows
   the window opening and painting from elisp.
4. No `target/` or compiled Rust is required for the demo (= the
   existing `src/main.rs` is untouched and unused by the spike).

If any of the unknowns in §3.3 turn out to be blockers (= GLib
threading, re-entrancy, or fundamental libffi/NeLisp incompatibility),
B0 *exits with findings* rather than retrying.  Doc 02 closes with a
written reason and Phase B is reconsidered.
